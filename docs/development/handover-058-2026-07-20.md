## What the feature does now

Scanning a folder writes an `inbox_source_groups` row and **no inbox item**. The
folder appears in the Inbox as a source-group row that is structurally
non-confirmable (no `inboxItemId` exists to hand `inbox.confirm`), carrying one
action: **Classify**. Classifying materialises the folder's real single-type
items, the group row disappears, and selection follows CHK011.

The hidden placeholder row this spec exists to delete is gone, along with the
read-side suppression that existed only to hide it.

## Status of the remaining items

### 1. Phase 6b — the needs-review split. **Already implemented.** (corrected)

An earlier draft of this handover said phase 6b "materialises ZERO siblings"
and was **wrong**. That conclusion came from probing `reclassify` (v1), which
is item-scoped and never calls `materialize_sub_items`. The path the UI
actually uses is `reclassify_v2`, and it splits correctly.

Verified end to end: classify a folder of unreadable headers (one needs-review
item), then supply per-file frame types plus the mandatory attributes each type
requires — exactly what "Apply manual overrides" sends. Result: **two
siblings**, `frame_type` `Some("light")` and `Some("dark")`, `needs_review = 0`
on both, no `mixed` item, and the pre-resolve item replaced rather than left
orphaned. That is T050's requirement, reusing the existing materialisation path
rather than a bespoke split.

Now pinned by `t052_needs_review_resolved_into_two_types_yields_two_siblings`
(T052). One subtlety the test documents: `materialize_sub_items` suppresses
`frame_type` while a group is still needs-review, so supplying frame types
*without* the mandatory attributes yields two siblings that both carry
`frame_type: None`. That is correct — a light frame with no OBJECT/FILTER/
EXPTIME is genuinely unresolved — but a test omitting them would assert a
weaker property than T050 states.

T051 (the item-identity hazard) is covered by T017's CHK011 selection plus the
pre-existing `pendingReclassifySelectionId` handoff; the detail pane is keyed
`sourceGroupId ?? inboxItemId`, so it survives the swap.

**The lesson worth keeping:** two entry points, one of which does not split by
design. `v1_reclassify_reports_an_aggregate_verdict_without_splitting` now pins
that distinction so the next reader does not repeat the mistake.

## 2026-07-21 — T051 root-caused, L3 7/7, main merged

### The defect: the handoff's navigate was clobbered by the cleanup

`inbox_ui_unclassified_gate_bulk_reclassify_unblocks_confirm` was the last
failing journey. Six earlier fix attempts all missed because they targeted the
wrong thing; the answer came from instrumenting `canConfirm`'s terms and the
`?selected=` URL in the same run:

```
21  items flip to new id   param=<old>   sel=NONE   old id gone, selection empty
22  HANDOFF decision=navigate -> <new>
23  param=<old>                                     navigate not yet applied
24  param=NONE                                      CLEARED, not set to <new>
```

`resolveReclassifyHandoff` was deciding correctly all along — it chose
`navigate` to an id that WAS present in the settled list. The bug was two
lines below it in `InboxPage.tsx`: `setPendingReclassifySelectionId(null)` ran
BEFORE `navigate`, and `navigate` is async. For one commit the guard was down
while `?selected=` still held the pre-split id, so `useStaleSelectionCleanup`
saw `selectedItem === undefined` with no pending handoff, opened its gate, and
its `selected: undefined` landed AFTER the handoff's navigate.

Fix: keep the pending id set across the navigate and clear it on the next run,
once `selected` already equals the target. Still bounded — the give-up path
clears immediately and a settled navigate falls through to the clear.

**L3 inbox journeys: 7/7** (was 6/7), all three SC-005 gates green.

### Cross-lane hazard hit during this session

The Real-UI harness loads the frontend from a SHARED `http://localhost:5173`
(`APP_URL`, baked into `tauri.conf.json`'s `devUrl`). Two lanes running E2E on
this machine therefore share one frontend: whichever `vite preview` is up
serves BOTH apps, so the second lane silently tests the first lane's build.

Observed live: this lane's preview (058 build) was serving another lane's
`--partition hash:4/4` run, whose `inbox_ui_journeys` then hung at >120s
because it was driving its own binary against 058's source-group row shape.

**There is no per-lane override** — `devUrl` is baked at build time. Local L3
runs must be serialized across lanes, or the port made configurable. Worth a
follow-up issue.
