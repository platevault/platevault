## What the feature does now

Scanning a folder writes an `inbox_source_groups` row and **no inbox item**. The
folder appears in the Inbox as a source-group row that is structurally
non-confirmable (no `inboxItemId` exists to hand `inbox.confirm`), carrying one
action: **Classify**. Classifying materialises the folder's real single-type
items, the group row disappears, and selection follows CHK011.

The hidden placeholder row this spec exists to delete is gone, along with the
read-side suppression that existed only to hide it.

## Status of the remaining items

### 1. Phase 6b — the needs-review split. **Already implemented.** (corrected)

An earlier draft of this handover said phase 6b "materialises ZERO siblings"
and was **wrong**. That conclusion came from probing `reclassify` (v1), which
is item-scoped and never calls `materialize_sub_items`. The path the UI
actually uses is `reclassify_v2`, and it splits correctly.

Verified end to end: classify a folder of unreadable headers (one needs-review
item), then supply per-file frame types plus the mandatory attributes each type
requires — exactly what "Apply manual overrides" sends. Result: **two
siblings**, `frame_type` `Some("light")` and `Some("dark")`, `needs_review = 0`
on both, no `mixed` item, and the pre-resolve item replaced rather than left
orphaned. That is T050's requirement, reusing the existing materialisation path
rather than a bespoke split.

Now pinned by `t052_needs_review_resolved_into_two_types_yields_two_siblings`
(T052). One subtlety the test documents: `materialize_sub_items` suppresses
`frame_type` while a group is still needs-review, so supplying frame types
*without* the mandatory attributes yields two siblings that both carry
`frame_type: None`. That is correct — a light frame with no OBJECT/FILTER/
EXPTIME is genuinely unresolved — but a test omitting them would assert a
weaker property than T050 states.

T051 (the item-identity hazard) is covered by T017's CHK011 selection plus the
pre-existing `pendingReclassifySelectionId` handoff; the detail pane is keyed
`sourceGroupId ?? inboxItemId`, so it survives the swap.

**The lesson worth keeping:** two entry points, one of which does not split by
design. `v1_reclassify_reports_an_aggregate_verdict_without_splitting` now pins
that distinction so the next reader does not repeat the mistake.

**ROOT CAUSE FOUND — it is not 058, and it is not the environment either.**

Two things were wrong, in sequence.

*First*, the local runner is missing what CI does. `.github/workflows/e2e.yml`
wraps the suite in `xvfb-run -s "-screen 0 1600x1200x24"` and sets
`WEBKIT_DISABLE_DMABUF_RENDERER=1` / `WEBKIT_DISABLE_COMPOSITING_MODE=1`.
`apps/desktop/scripts/run-e2e-real.sh` does NEITHER — it calls `cargo nextest`
directly with no display wrapper and no WebKit env. Supplying them changes the
failure, which is how you know it was real. (The screen size matters too: CI's
comment notes the 1280 default is narrower than the 1400px viewport journeys
request, and a clamped resize yields "a journey that passes while asserting
nothing.")

*Second*, and the actual blocker: with xvfb + WebKit env supplied, the error
becomes `bridge_ready script failed: Window not found: no such window`. The app
itself is healthy — its own log shows `window position was off-screen;
recentered` and `warmed 13073 bundled target seed entries`. What changed is the
window topology. `tauri.conf.json` now declares TWO windows:

```
main   -> visible: false      (hidden until the app is ready)
splash -> url: splash.html
```

That is main's branded splash screen (#1248/#1299, landed 2026-07-20). On
launch the only visible window is the splash; the WebDriver session attaches to
it; the splash closes when the app finishes booting; every later probe gets
`no such window`. The harness never switches windows — the one
`["main", "splash"]` reference in `common/mod.rs:2158` is Windows localStorage
cleanup, not window selection.

This breaks EVERY journey in the repo, which is exactly what was measured: 18
failed, and the only 8 passes were the per-binary
`webview_subdir_matches_the_app` check that needs no app.

**Fix direction** (not attempted here — it is harness work owned by the L3
lane, not 058): after connecting, enumerate window handles and switch to the
`main` webview rather than accepting the default, and wait for it to become
visible. Alternatively gate the splash off under the `e2e` feature.

**What this means for the gate:** L3 now RUNS and has caught a real 058 defect.
That is the gate working, not the gate broken.

After the two harness fixes above, the journey drives the real app and fails at:


The list shows **"no detections."** — its empty state.

**The data is there.** `inbox.list`, invoked through the bridge against the SAME
running app at the moment of failure, returns the group:

```json

  {"fileCount":2,"format":"fits","lane":"move","relativePath":"", ...}]}
```

So the backend is correct and the UI drops it. The failure path now prints both
views side by side, so this is one run to reconfirm, not several.

Eliminated by inspection, do not re-check:
- `commands.inboxList()` takes no args — not an argument-shape problem.
- `kindFilter` defaults to `''`, and the guard is `if (kindFilter && kindFilter !== 'all')` — falsy, so it does not filter.
- `filterType` is `'all'` — `sourceGroupLane` filtering does not run.
- `filteredSourceGroups` (InboxPage) returns `sourceGroups` unchanged when the search box is empty.
- The served bundle DOES contain `inbox-source-group-classify-` (checked in `dist/assets/InboxPage-*.js`) — not a stale build.
- `sub_frame_count` is 2, so `file_count > 0` passes in `list_unclassified_source_groups`.

**Where to look next:** `InboxList` reaches `visualRows.length === 0`, which is
what renders the empty state — so the source-group rows are not entering
`visualRows` even though nothing in the filter chain removes them. Start at the
`visualRows` construction (`InboxList.tsx` ~494) and at whether
`listData.sourceGroups` survives `unwrap()`/the generated binding at runtime.

**Why no unit test caught it:** every existing test passes `sourceGroups` in as
a prop. None exercises the real `useInboxList` → `InboxPage` → `InboxList` path,
which is exactly the seam that is broken. A regression test should drive the
page with a mocked `commands.inboxList` returning a source group, not hand the
list a prop.

## Traps worth not re-learning

- **Green tests proved nothing for T012.** Deleting the placeholder left all
  2528 workspace tests passing — `inbox_scan_folder` had *zero* Layer-1 coverage
  of that behaviour. Two L1 tests were added; L3 remains the only end-to-end
  check.
- **A vacuous assertion nearly shipped.** "No row from `list_inbox_sub_items` has
  an empty `group_key`" cannot fail — that query filters `group_key != ''` in
  SQL. Compare the *unfiltered* count instead.
- **testid prefix collision.** `inbox-source-group-<id>` is a strict prefix of
  `inbox-source-group-classify-<id>`, and `testid_suffix` returns the first
  match. Derive ids from the button prefix.
- **Never chain a gate and a commit in one command.** `clippy | grep -c` piped
  into a `git commit` let a `-D warnings` failure ship (`b1348161`), because
  `cargo check` and `nextest` stay green on an unused import.
- **`-p app_core` is not `-p app_core_inbox`.** Targeted runs against the wrong
  crate reported 480 passing tests that never included the code under change.
- **Probe the path the product uses.** Concluding phase 6b was unimplemented
  came from testing `reclassify` (v1) when the UI calls `reclassify_v2`. Two
  entry points, only one of which splits — the wrong one produced a confident,
  wrong finding that reached a written handover.
