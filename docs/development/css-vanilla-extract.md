# CSS: vanilla-extract authoring guide

## Overview

Styles in the desktop app migrate from hand-authored BEM CSS files to
[vanilla-extract](https://vanilla-extract.style/) (VE). VE generates
zero-runtime, type-safe CSS with hashed class names at build time.

The migration is per-component. Until a component is migrated, its BEM CSS
file remains active. Both systems coexist — no existing styles break on
adoption.

## File layout

Each component gets a co-located `.css.ts` file:

```
src/app/
  LogPanel.tsx
  logpanel.css.ts     ← VE styles for LogPanel
  LogEntryRow.tsx     ← consumes exports from logpanel.css.ts
```

Place the `.css.ts` file in the same directory as the component it styles.
Do not create shared multi-component style files; name collisions and
implicit coupling follow.

## Imports: vars is mandatory

Every `.css.ts` file must import `vars` from the theme contract:

```ts
import { style, styleVariants } from '@vanilla-extract/css';
import { vars } from '@/styles/themes.css';
```

Use `vars.<token>` for every color, border, or shadow that must respond to
theme switching. Do not write raw `var(--pv-*)` strings inside `.css.ts` files
— the TypeScript compiler cannot catch a typo in a raw string, but it will flag
a missing key on `vars`.

For theme-invariant tokens (spacing, typography, radii, weights, and layout
dimensions), import `uvars`:

```ts
import { style } from '@vanilla-extract/css';
import { vars } from '@/styles/themes.css';
import { uvars } from '@/styles/themes.css';

export const btn = style({
  height: uvars.controlH,
  padding: `0 ${uvars.sp3}`,
  borderRadius: uvars.radiusSm,
  color: vars.text,
});
```

`uvars` maps to `createGlobalTheme(':root', ...)` and covers all keys from
`tokens/foundation.json` and `tokens/component/*.json`. Use `uvars.*` instead
of raw `var(--pv-sp-*)`, `var(--pv-text-*)`, `var(--pv-radius-*)` etc. — the
same typo-safety guarantee applies.

## styleVariants for modifiers

Use `styleVariants` when a set of variants shares a base style and differs only
in one property family. Key the variants on a TypeScript union type so the
compiler enforces exhaustiveness:

```ts
// ✓ — key matches the LogEntry['level'] union
export const levelVariants = styleVariants({
  info:  { ...levelBase, color: vars.info },
  warn:  { ...levelBase, color: vars.warn },
  error: { ...levelBase, color: vars.danger },
  debug: { ...levelBase, color: vars.textMuted },
});

// in the component:
<span className={levelVariants[entry.level]}>…</span>
```

Do not use `styleVariants` for layout variants or structural differences. Use
separate named `style()` exports for those.

## globalStyle escape hatch

`globalStyle` is permitted only for:

1. Selectors that cross a component boundary — e.g. a child inside a
   third-party component root that cannot accept a `className` prop.
2. The generated theme file (`src/styles/themes.css.ts`) written by
   `gen-ve-themes.mjs` — do not edit this file by hand.

Do not use `globalStyle` to style elements that could receive a VE class
directly.

## Theme contract

`src/styles/themes.css.ts` is generated. Do not edit it by hand.

Regenerate after changing token sources:

```sh
pnpm --filter @astro-plan/desktop ve:themes
# or:
just ve-themes-check    # drift check only (CI gate)
```

The file uses `createThemeContract` over 49 semantic keys and
`createGlobalTheme('[data-theme=X]', …)` for each of the 6 themes. Theme
switching at runtime is a `data-theme` attribute flip on the document root —
no JavaScript bundle change required.

The 6 themes are: `espresso-dark`, `observatory-cool-light`,
`observatory-cool`, `observatory-dark`, `warm-clay`, `warm-slate` (default).

## CI gates

Two checks run on every PR in the `lint` step:

| Check | Command |
|---|---|
| Token drift (DTCG → `tokens.css`) | `node scripts/build-tokens.mjs --check` |
| VE theme drift (DTCG → `themes.css.ts`) | `node scripts/gen-ve-themes.mjs --check` |

Both checks diff the generated file against the committed file. A non-zero
diff fails CI. Regenerate and commit when token sources change.
