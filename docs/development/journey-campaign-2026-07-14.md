## Decisions

- **D1 — Init in place.** `docs/product/journeys/` already held a
  pre-format journey catalog (J01–J17, prose "Stage" style). Initialized
  the FORMAT scaffold into that directory rather than a new one — one
  journeys dir per product; this is it.
- **D2 — Legacy INDEX preserved.** The old `INDEX.md` was a hand-written
  Wave-0 rerun plan, not a routing table. Renamed to `wave0-rerun-plan.md`;
  `INDEX.md` is now generated by `journeys.py`.
- **D3 — Reporter = github-issues**, labels `bug, phase:build` plus the
  existing `journey-<n>` label per finding. Derived from the invocation
  ("issues per user journey in github") and the live label taxonomy.
- **D4 — fix_loop = dispatch-coder, max 3** (invocation: "revert any
  issues back to the coder"). runs_keep = 20 (default, not user-confirmed).
- **D5 — Migration now, all 17 journeys.** The campaign's verify/write
  steps require FORMAT-conformant journeys (step ids, frontmatter, Δ log),
  so migration is a prerequisite, inserted before the fix wave.
  journey-scribe agents rewrite each journey as current truth, folding the
  2026-07-14 `deltas/` docs in where the underlying change is actually
  merged; `trace:` links the legacy doc's git sha. Legacy `deltas/` and
  wave0 sheets are kept as historical evidence (pointer-stubbed in README).
- **D6 — Migrated journeys start `status: active`, `version: 1`.** They
  document long-standing, previously validated behavior (validation runs
  through 2026-07-14 exist); `draft` is for never-validated journeys. New
  journeys authored in Phase 5 start `draft` per FORMAT.
- **D7 — Fix-wave scope = open issues carrying a `journey-<n>` label**
  (~66 of 249 open issues), grouped per journey, split into coder-sized
  nodes with disjoint file scopes. Explicitly OUT of scope: the ~64
  `spec-030` backlog issues and other spec-only/epic/enhancement issues
  with no journey label — those belong to SpecKit lanes, not this campaign.
  Cap documented here per the no-silent-caps rule.
- **D8 — Issues owned by the six in-flight branches are excluded** from my
  coder wave, and my nodes' scopes avoid those branches' files where
  predictable, to keep the gatekeeper's merges clean.
- **D9 — Verification gate.** All journey-validator runs wait until the six
  in-flight branches merge (owner instruction). Doc-only phases (migration,
  journey-write authoring) proceed meanwhile; first-validation-fresh-context
  rule is respected by never letting the authoring scribe validate its own
  journey.
- **D10 — Single validation profile.** `windows-desktop` (real Tauri app
  via Tauri MCP bridge), `exclusive: true` → validators run serially. The
  mock/Vite runtime is not a validation target (fakes the backend).
- **D11 — Branch/PR discipline.** Journeys-dir waves go via short-lived
  `journey/*` branches + squash PRs; code fixes via per-node
  `coder/*` branches + PRs, CI-shepherded, merged FCFS by the gatekeeper.
