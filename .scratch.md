## Reused infra found
- crates/app/inbox/src/target_recommendations.rs (spec 041 R-17): ALREADY implements
  per-light-frameset pointing load (inbox_file_metadata ra_deg/dec_deg/focal_length_mm/
  pixel_size_um/naxis1/2/sky_rotation_deg/rotator_angle_deg), rotation-aware FOV field
  membership via target_match::{rank,Constraint,SkyObject}, fixed-radius fallback. This
  is exactly T028's rotation-aware footprint machinery for LOCAL canonical_target
  candidates. P3 cone-search should reuse Constraint/rank the same way, sourcing
  candidates from simbad_resolver::PositionResolver::resolve_position (online TAP cone
  search) instead of (in addition to) the local cache.
- simbad-resolver 0.2.0 (already bumped by P1): TapResolver implements PositionResolver
  (resolve_position(ra,dec,radius,limit) -> Vec<PositionMatch>), CIRCLE/CONTAINS ADQL,
  ordered nearest-first. IMPORTANT: resolve_position's assemble_identity call passes
  `Vec::new(), None` for aliases/common_name (tap.rs:238) — cone-search hits do NOT
  carry common_name/aliases (no second alias round-trip, by design, for perf). To honor
  OQ-1 (common_name promotes top tier) + OQ-2 (retain named stars), each top-N candidate
  needs a follow-up `Resolver::resolve(primary_designation)` call to enrich common_name/
  aliases — bounded to top-N, acceptable.
- astro-plan's local targeting_resolver::ResolvedIdentity (crates/targeting/resolver/
  src/lib.rs) intentionally DROPS otype_raw (comment: "nothing in this codebase reads
  it"). OQ-2's star-family exclusion needs otype_raw (many single-star SIMBAD otypes
  like `*`,`V*`,`PM*`,`HB*`,`WR*`,`Em*` fall into ObjectType::Other via map_otype, so
  ObjectType alone can't distinguish a star from a nebula lumped into Other). Plan: do
  the OQ-1/OQ-2 ranking directly over simbad_resolver::ResolvedIdentity (upstream crate
  type, which HAS otype_raw + common_name), not astro-plan's stripped local type —
  convert to local types only at the confirm/persist boundary (existing
  targeting_resolver::simbad::from_crate_identity).

## BLOCKING GAP FOUND: no WCS/CRVAL extraction anywhere in the codebase
- FR-012 requires: WCS CRVAL1/2 (plate-solved, HIGH conf) -> mount OBJCTRA/OBJCTDEC
  (MEDIUM conf) -> none. Grepped crates/metadata/fits, crates/metadata/xisf,
  crates/metadata/core, and the whole repo for CRVAL/CTYPE/CRPIX/WCS: zero hits.
- crates/metadata/fits/src/lib.rs:170-172 and crates/metadata/xisf/src/lib.rs:143-147
  populate MetadataExtracted.ra_deg/dec_deg ONLY from decimal `RA`/`DEC` header or
  sexagesimal `OBJCTRA`/`OBJCTDEC` fallback — both are MOUNT-reported pointing, never a
  WCS plate-solve. inbox_file_metadata (migration 0049) has one ra_deg/dec_deg pair, no
  wcs vs mount distinction.

  high-confidence" criterion cannot be met by real data without: (a) new CRVAL1/CRVAL2 (+
  likely CTYPE/PLTSOLVD sanity check) extraction in metadata/fits + metadata/xisf, (b) a
  new inbox_file_metadata migration adding wcs_ra_deg/wcs_dec_deg columns (or a
  pointing_source enum), (c) precedence logic in the pointing-derivation step.
