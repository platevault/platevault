# Data Model: Catalog Index Licensing

**Spec**: 014-catalog-index-licensing | **Plan**: [plan.md](./plan.md)

This data model covers the registry and attribution surface only. The
minimal catalog entry shape (`name, identifiers, ra, dec, source`) is
owned by spec 013 and is referenced but not redefined here.

## Entities

### Catalog

Represents one registered catalog index known to the app.

| Field          | Type                       | Notes                                                   |
| -------------- | -------------------------- | ------------------------------------------------------- |
| `id`           | `string` (slug)            | Stable identifier, e.g. `messier`, `ngc`, `ic`.         |
| `name`         | `string`                   | Human-readable display name.                            |
| `version`      | `string` (semver-ish)      | Bundle version. Bumped by CI on every regeneration.     |
| `license`      | `string` (short code)      | e.g. `public-domain`, `cc-by-4.0`, `apache-2.0`.        |
| `origin`       | `enum`                     | `built_in` or `user`.                                   |
| `source_url`   | `string` (URI)             | Upstream source the bundle was generated from.          |
| `last_updated` | `Timestamp` (RFC 3339 UTC) | When the bundle was regenerated by CI (or registered).  |
| `entry_count`  | `integer` (>= 0, optional) | Number of entries in the index. Surfaced in Settings.   |

Invariants:

- `id` is unique across the registry.
- `built_in` catalogs are seeded from the compiled-in manifest and
  cannot be removed at runtime.
- `user` catalogs may be added/removed but inherit the same field set.
- Every `Catalog` MUST have at least one matching `LicenseAttribution`
  record (even if the license is `public-domain`).

### LicenseAttribution

Represents the per-catalog notice surface used in Settings and in
generated NOTICE files for downstream redistribution.

| Field         | Type                       | Notes                                                  |
| ------------- | -------------------------- | ------------------------------------------------------ |
| `catalog_id`  | `string` (slug)            | FK to `Catalog.id`.                                    |
| `text`        | `string` (non-empty)       | Full required notice text, verbatim.                   |
| `link`        | `string` (URI)             | Source link the notice references.                     |
| `license`     | `string` (short code)      | Echoes `Catalog.license` for caller convenience.       |
| `accessed_on` | `Timestamp` (optional)     | Date the source was fetched (used for public-domain    |
|               |                            | "verified on" entries).                                |

Invariants:

- `text` is never empty. Public-domain catalogs still carry a
  "verified: <source>, accessed <date>" string so users can audit the
  declaration.
- `link` MUST resolve to a stable source URL or an archived snapshot
  (e.g. Internet Archive). Mirror lists are out of scope for v1.
- One `Catalog` MAY have multiple `LicenseAttribution` rows when a
  bundle aggregates more than one upstream source; the panel renders
  them grouped under the catalog.

## Relationships

```
Catalog 1 ───< LicenseAttribution  (by catalog_id)
```

No other entity references either record in v1. Spec 013's target
lookup pipeline reads catalog index files directly via the registry
and does not embed catalog metadata into target records.

## Storage

- Built-in catalogs: compiled-in manifest (generated by CI from the
  catalog source definitions). Manifest is loaded into memory at app
  start; not stored in SQLite.
- User-added catalogs: SQLite table `catalog_user(id, name, version,
  license, source_url, last_updated, entry_count)` plus a paired
  `catalog_user_attribution(catalog_id, text, link, license,
  accessed_on)` table.
- The `Catalog` and `LicenseAttribution` records returned by the
  contracts are the union of both sources.

## Out of Scope

- Catalog entries themselves (owned by spec 013).
- Update / refresh transactions (deferred to a future update spec; see
  research R3).
- Cross-catalog identifier reconciliation (no `Catalog` row references
  another `Catalog` row).
