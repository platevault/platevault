## US 2 — Onboard An Existing Project (P2)

- [ ] US2-1..US2-7: RE-VERIFIED, stays deferred — checked `docs/design/` for a
  since-added onboard-existing-project mockup; none exists (the only
  "onboarding" wireframes found are first-run wizard variants, a different
  flow). Design pass genuinely still required before implementation; not a
  reconciliation candidate. **Deferred.** Reason: no mockup precedent exists and
  design research R6 (three-way marker reconciliation) has not been validated
  against a real library root. The building blocks are in place:
  - `project_structure::parse_marker()` handles the three-way reconcile logic
    (no marker → propose write; parsable → recover; unparsable → refuse).
  - `project.create` accepts pre-existing paths (path.collision error guards it).
  - The folder picker (spec 004 `native_directory_pick`) is available.
  The onboard wizard surface (US2-2 through US2-6) requires a design pass
  before implementation. Tracked as a follow-up to spec 008. A minimal
  functional path would be: folder pick → call `parse_marker` → if parsable
  show recovered metadata and call `project.create` → if no marker show the
  create dialog pre-filled with the detected path.

## Out of Scope

- Multi-source bulk add (current contract is single-session per call;
  callers loop for batch add).
- Tool change after `prepared` lifecycle (blocked by `tool.locked`;
  recovery is manual re-creation via `project.create` — R-NoDup; no
  `project.duplicate` in v1).
- Templated name suggestions (research R5 alternative, deferred).
- Project rename → folder rename (current rename is metadata-only;
  filesystem rename is a follow-up plan).
- Playwright / visual smoke tests (WSL headless, no Tauri dev server available
  in this environment — all functional logic verified via vitest + in-memory
  Rust tests instead).
