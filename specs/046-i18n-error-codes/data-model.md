# Data Model: i18n Infrastructure & Unified Error-Code Translation

**Feature**: `046-i18n-error-codes` | **Date**: 2026-06-22

This feature is presentation/contract plumbing; it introduces no persisted data
and no database schema change. The "entities" are compile-time artifacts.

---

## E1 — Message catalog

The single source of user-facing wording.

| Field | Type | Notes |
|-------|------|-------|
| key | stable identifier (snake_case, area-prefixed) | Valid JS identifier; see R5 scheme. Unique. |
| text | English string | Post-cleanup wording (FR-014); no dev-status markers (FR-012). |
| params | named placeholders | e.g. `{count}`, `{name}`, `{path}`; typed at the call site (FR-003). |

- **Storage**: `apps/desktop/messages/en.json` (`{ "<key>": "text {param}" }`).
- **Projection**: compiled to typed functions in `apps/desktop/src/paraglide/`
  (git-ignored, regenerated). Access: `m.<key>(params?)`.
- **Invariants**:
  - Referencing a non-existent key fails the build (FR-002).
  - Adding `locales` (e.g. `de.json`) requires no call-site change (FR-005).
  - English renders synchronously, offline, no flash (FR-004).

### Key namespaces (taxonomy)

| Prefix | Concept |
|--------|---------|
| `common_*` | shared atoms (ellipsis, yes/no, cancel, close) |
| `shell_*` / `nav_*` | app shell, navigation, status bar |
| `inbox_*`, `sessions_*`, `calibration_*`, `targets_*`, `projects_*` | feature areas |
| `settings_*`, `setup_*` | settings panes, first-run wizard |
| `verb_*` | canonical action verbs (US3): `verb_archive`, `verb_trash`, … |
| `err_*` | error-code messages + `err_generic_fallback` |

## E2 — Error-code registry (authoritative; pre-existing)

The enumerated set of machine-readable failure codes. **Single source of truth**,
already implemented.

| Field | Type | Notes |
|-------|------|-------|
| variant | Rust enum variant | `ErrorCode::PathNotExists`, … (~95) |
| wire string | `#[serde(rename="…")]` | Byte-exact dotted code, e.g. `path.not_exists` |

- **Backend**: `crates/contracts/core/src/error_code.rs` (`enum ErrorCode`,
  derives `Serialize/Deserialize/Type`).
- **Contract projection**: `apps/desktop/src/bindings/index.ts` →
  `export type ErrorCode = "validation.request_envelope_invalid" | … ;`
  (generated by tauri-specta; regenerated by `tests/bindings.rs`; CI diff-checked).
- **This feature's change**: `ContractError.code` is typed `ErrorCode` (was
  `String`) so an unregistered code cannot be emitted (FR-006) and the union
  flows unbroken to the frontend (FR-007).
- **Invariant**: backend and frontend share exactly one code set → zero drift
  (SC-004).

## E3 — Error translation map (single point)

Maps an error code to a catalog message, with a typed fallback.

| Field | Type | Notes |
|-------|------|-------|
| code | `ErrorCode` | key of the map |
| message-key | catalog message fn | `err_<code>`; resolves via `m.*()` |
| fallback | catalog message fn | `err_generic_fallback` when code unmapped |

- **Location**: `apps/desktop/src/lib/error-messages.ts` (the map) +
  `apps/desktop/src/lib/errors.ts` (`errMessage()` — the *only* place a thrown
  value becomes user text).
- **Behavior**:
  - Known code → `m.err_<code>(details?)` (FR-008).
  - Unknown/unmapped code → `m.err_generic_fallback()` **and** log the raw code
    internally (FR-010, FR-011, SC-005).
  - Never returns the raw code or backend exception string (FR-009).
- **Type relationship**: the map is keyed by `ErrorCode`; adding a backend code
  without a mapping is detectable at type-check (FR-007, SC-003). Implementation
  may use `Partial<Record<ErrorCode, …>>` + explicit fallback, paired with a
  type-level exhaustiveness check (a `satisfies` assertion or a unit test that
  iterates the union) so the gap is caught before release.

## Relationships

```
ErrorCode (Rust enum)  ──specta──▶  ErrorCode (TS union)  ──keys──▶  Error translation map
   (E2, authoritative)                  (E2 projection)               (E3, single point)
                                                                          │ resolves to
                                                                          ▼
                                                                   Message catalog (E1)
                                                                          ▲ also feeds
   every UI component string ──────────────────────────────────────────┘
```

One catalog (E1) is the terminal sink for **all** user text — both ordinary UI
strings and the messages that error codes (E2) resolve to through the single
translation map (E3).
