# Research: i18n Infrastructure & Unified Error-Code Translation

**Feature**: `046-i18n-error-codes` | **Date**: 2026-06-22

Phase 0 decisions. Each records the choice, the rationale, and the alternatives
rejected, per Constitution principle IV (Research-Led Domain Modeling).

---

## R1 — Message catalog library: Paraglide JS (`@inlang/paraglide-js` 2.x)

**Decision**: Adopt Paraglide JS (inlang) as the message catalog, integrated via
its Vite plugin. Messages live in `apps/desktop/messages/en.json`; the compiler
emits type-safe message functions to `apps/desktop/src/paraglide/` consumed as
`import { m } from '@/paraglide/messages'` → `m.some_key()`.

**Why it fits this spec's hard requirements**:

- **Build-time, type-safe (FR-002, SC-003)**: messages compile to individual
  functions. Referencing a non-existent key is a TypeScript/compile error, not a
  runtime blank or raw-key render. This is the single most important property and
  the reason a *compiled* catalog beats a runtime JSON-lookup library.
- **Interpolation is type-checked (FR-003)**: a message with `{count}` compiles to
  a function whose argument object is typed; values are passed, never concatenated.
- **Offline / synchronous / no flash (FR-004, US1 AS3)**: compiled functions are
  plain JS, bundled into the app. No async resource fetch, no network, no loading
  state — exactly right for a local-first Tauri app.
- **Multilingual-ready without touching components (FR-005, SC-007)**: adding a
  locale = adding `messages/<locale>.json`; component call sites
  (`m.some_key()`) are unchanged. The throwaway SC-007 proof is "add `de.json`,
  confirm a screen flips with zero component edits."
- **Tree-shakeable**: unused messages drop from the bundle.

**Version**: `@inlang/paraglide-js` 2.20.x (latest stable as of 2026-06-22).

**English-pinned, no switcher (FR-004)**: `project.inlang/settings.json` sets
`baseLocale: "en"`, `locales: ["en"]`. The runtime locale is fixed to `en`
(single-locale strategy / `baseLocale` only); no locale-detection strategy
(cookie/url/navigator) is enabled, and no `setLocale` UI is shipped. Result:
the app always renders English, deterministically, with no switcher surface.

**Alternatives rejected**:

- **`react-i18next` / `i18next`**: runtime JSON lookup. Missing keys fall back to
  the key string at *runtime* (violates FR-002/SC-003 unless bolted onto extra
  type-gen tooling), pulls a heavier runtime, and encourages async resource
  loading (flash risk). Type-safety is add-on, not intrinsic.
- **`react-intl` (FormatJS)**: ICU-first, runtime formatting, provider/context
  overhead, weaker compile-time key guarantees without extra extraction tooling.
- **Hand-rolled `Record<Key, string>` + `t(key)`**: gives type-checked keys but
  no compiler, no interpolation typing, no tree-shaking, and re-implements
  pluralization/locale plumbing we'd want for FR-005. The existing
  `ERROR_MESSAGES` map is exactly this and is being *folded into* the catalog,
  not generalized app-wide.
- **Lingui**: macro-based, viable, but Paraglide's compiled-function model gives
  stronger missing-key-at-build guarantees with a lighter footprint and a
  first-class Vite plugin. (This library was the pre-made decision in the feature
  brief; research confirms it satisfies every FR.)

**Risks**: Paraglide owns a compiled output dir — must be git-ignored and
regenerated by the Vite plugin on dev/build (and in CI before typecheck). The
`m.*` import path must resolve in `tsconfig`. Mitigated in US1 infra task.

## R2 — Error-code registry: reuse `contracts_core::error_code::ErrorCode`

**Decision**: Do **not** introduce a new registry. The authoritative enumerated
set already exists: `crates/contracts/core/src/error_code.rs` defines
`ErrorCode` (~95 variants, each with an explicit `#[serde(rename = "wire.string")]`),
derives `specta::Type`, and is exported to `apps/desktop/src/bindings/index.ts`
as the `ErrorCode` string union by the tauri-specta builder
(`desktop_shell::specta_builder()`), regenerated by
`apps/desktop/src-tauri/tests/bindings.rs` on `cargo test` and diff-checked in CI.

This **is** the language-neutral, drift-proof contract FR-006/FR-007 require. It
landed as additive scaffolding in spec 042 (T011).

**Remaining work for this feature** (the part 042 deferred):

- Tighten `ContractError.code` from `String` to `ErrorCode` at the type level so
  the backend cannot emit an unregistered code and the frontend receives the
  exact union (FR-006). (042 left `code: String` deliberately.)
- Make the frontend translation **exhaustive-aware**: the code→catalog-key map is
  typed `Record<ErrorCode, MessageKey>` (or `Partial` + a typed fallback) so that
  adding a backend code surfaces as a type gap before release (FR-007, SC-003).

**Alternatives rejected**:

- **New TS-only error enum**: would immediately violate FR-007 (drift) — two
  sources of truth.
- **Keep `code: String` forever**: loses the compile-time exhaustiveness that
  SC-003/SC-004 demand; the union exists, so use it.

## R3 — Single translation point: extend existing `@/lib/errors`

**Decision**: The frontend already centralizes error→message resolution in
`apps/desktop/src/lib/errors.ts` (`errMessage()` / `asError()` / `isContractError()`),
backed by `lib/error-messages.ts` (`ERROR_MESSAGES: Partial<Record<ErrorCode, string>>`).
Nearly all call sites already import `errMessage` from `@/lib/errors` rather than
re-implementing mapping. So FR-008's "single translation point, remove
per-component mappers" is **mostly already true**; this feature finishes it:

1. Repoint `ERROR_MESSAGES` values from inline English literals to **catalog
   keys** (`m.err_path_not_exists()` …), so error wording lives in the one catalog
   like everything else (US1/US2 convergence).
2. Add the explicit generic fallback message (catalog key) + structured logging of
   the unmapped code at the single point (FR-010, FR-011, SC-005).
3. Sweep for any residual ad-hoc `err.message` / `String(err)` / inline
   code-switch in components and route them through `errMessage`.

**Why not a React context / hook for errors**: unnecessary — error translation is
a pure function of `(thrown value) → string`; a module-level helper is simpler,
testable without rendering, and already adopted.

**`details` payload (edge case)**: `ContractError` carries optional
`details` (e.g. `{ dimensions }`). The translation layer MAY consume `details` to
build a richer message via an interpolated catalog entry, without exposing the raw
structure (FR's edge-case + assumption). Handled per-code where it adds value;
default path ignores `details`.

## R4 — Lint gate: ESLint rule forbidding hardcoded user-facing literals

**Decision**: Add a project ESLint rule (flat config, `eslint.config.js`) that
flags hardcoded user-facing strings, wired into `just lint` alongside the existing
`check-tokens.sh` guard. The rule is the **engine of the migration**: it makes
un-catalogued strings a lint failure, so the migration is "drive the rule to
zero," and it prevents regressions afterward (FR-001, FR-012, SC-001, SC-002).

**Scope of what counts as "user-facing"** (rule targets):

- JSX text node children that contain a letter (`<button>Save</button>`).
- A fixed set of JSX attributes: `placeholder`, `title`, `aria-label`,
  `aria-description`, `alt`, and `label`/`heading`-style props on known shared
  components.
- String args to toast/notify helpers.

**Out of scope (allowed literals)** — to keep false positives down:

- Non-letter strings (punctuation-only, numbers, `·`, paths, hex).
- `className`, `id`, `data-*`, `key`, `type`, `role`, `href`, `to`, enum/discriminant
  values, and other machine strings.
- Catalog source itself, generated `bindings/`, `paraglide/`, tests, fixtures,
  mocks, and `dev/` developer-mode surface (excluded; per spec, tests/fixtures are
  out of SC-001's scope).
- An inline `// eslint-disable-next-line alm/no-user-string` escape hatch with a
  required reason, plus a small documented allowlist for genuine edge cases.

**Implementation choice**: prefer a **local custom rule** (a tiny rule module in
`apps/desktop/eslint-rules/`) over a third-party plugin, because the existing repo
already ships a bespoke lint guard (`check-tokens.sh`) and a custom JSX
no-inline-style rule precedent (spec 028, task #11), and a local rule lets us tune
exactly which attributes/props are user-facing for *this* component library.
Evaluate `eslint-plugin-i18next`'s `no-literal-string` as a reference for edge
cases, but a local rule avoids dragging i18next semantics in.

**Rollout to avoid a 200-file red wall**: introduce the rule, then either (a)
enable per-directory as each area is migrated, or (b) ship a baseline/allowlist of
known-unmigrated files that shrinks each wave. Chosen: per-area enablement via
`overrides`/`files` globs so `just lint` stays green between waves while still
gating already-migrated areas. SC-001 is met when the rule is on for `src/**`
(minus the documented exclusions) with zero violations.

## R5 — Catalog key naming scheme

**Decision**: Flat, snake_case, namespaced-by-area keys with a small fixed
taxonomy. Paraglide keys are flat identifiers (valid JS identifiers), so namespace
via prefix, not nesting.

```
<area>_<element>_<intent>
```

- Area: `inbox`, `sessions`, `calibration`, `targets`, `projects`, `settings`,
  `setup`, `shell`, `common`.
- Element/intent: `..._title`, `..._empty`, `..._btn_save`, `..._ph_search`
  (placeholder), `..._tip_*` (tooltip), `..._toast_*`, `..._confirm_*`.
- Errors: `err_<code-with-dots-as-underscores>` (e.g. `err_path_not_exists`),
  plus `err_generic_fallback`.
- Shared vocabulary (US3, FR-013, SC-006): one canonical key per concept —
  `verb_archive`, `verb_trash`, `nav_my_targets`, `common_ellipsis` — referenced
  everywhere, so a concept is named once.

**Rationale**: prefixing keeps related strings discoverable and greppable, sidesteps
Paraglide's flat-key constraint, and makes "one canonical term per concept"
auditable (search `verb_*`, `nav_*`). Interpolation params use the names in the
spec's edge cases (`count`, `name`, `path`).

**Alternative rejected**: dotted/nested keys — not supported as call identifiers by
the compiler; would force a different access pattern and lose type-safe direct
calls.

## R6 — Preserving wording & vocabulary consistency (FR-013, FR-014, US3)

**Decision**: Migration is mechanical: lift the **exact** current post-cleanup
English into catalog entries (FR-014). Where the audit finds synonyms for one
concept (destructive verbs `archive`/`trash`, the "My Targets"/favourites view,
section titles, ellipsis `…` vs `...`), collapse to a single canonical key (US3)
as an **explicit, isolated** change with the wording recorded in the catalog —
never silently during a mechanical lift. The catalog becomes the review gate that
also enforces FR-012 (no STUB/MOCK/"coming soon" leakage), since the copy-cleanup
already landed (ef364df) and new leakage would appear as a reviewed catalog diff.
